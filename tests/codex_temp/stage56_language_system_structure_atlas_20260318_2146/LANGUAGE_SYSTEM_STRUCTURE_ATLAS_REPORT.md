# 语言系统结构总汇图

- 核心公式: `h_language = B_anchor + F_modifier + A_relation + G_control + C_closure`
- 系统主张: 语言系统更像分层锚点、修饰纤维、关系轴、控制轴和闭包读出的组合结构，而不是单一连续词向量场。

## 已有证据支持的结构
- 名词/概念: 普通名词和具体概念更像家族锚点上的局部实例偏移，而不是孤立向量点。
- 形容词: 形容词型微观属性更像附着在实体锚点周围的修饰纤维，而不是独立实体锚点。
- 关系轴: king / queen 这类关系不是全局线性空间，而是局部关系轴上的可迁移偏移。
- 生成控制: style / logic / syntax 不是概念本体，而是并行控制轴；它们决定概念在生成时怎样被读出、约束和闭包。

## 词类结构
- 名词: 普通名词更像锚点加实例偏移。
- 形容词: 形容词更像挂在锚点上的修饰纤维。
- 动词: 动词或动作词更像后继传输或协议操作子，而不是静态实体锚点。
- 抽象名词: 抽象名词更像协议束或关系束，不像纯实体锚点。
- 副词: 副词更可能是二阶调制子：它主要改写动作路径、强度和读出方式，而不是提供新的实体锚点。（推断）

## 为什么会长出这种结构
- 共享压缩: 大量词项共享家族统计，最省参数的办法不是给每个词单独建模，而是先学家族基底，再学实例偏移。
- 修饰复用: 形容词之所以长成纤维结构，是因为红、甜、重这类修饰要能在很多名词上重复挂接，最自然的实现就是可迁移修饰纤维。
- 关系局部线性: king / queen 这类局部类比之所以出现，是因为模型在很多上下文里重复看到稳定角色变换，最终形成局部关系偏移轴。
- 生成多约束: style / logic / syntax 必须同时满足，所以网络会长出并行控制轴，而不是把所有生成约束压成一条单轴。
- 闭包压力: prototype / instance / union 之所以会形成 stage6 闭包结构，是因为语言系统不仅要存词义，还要在上下文里把词义拼成可执行读出。

## 最严格的硬伤
- 副词目前没有单独因果探针，当前只能做结构推断，不能当成已证事实。
- relation axis 目前仍主要来自 king / queen 这类局部例子，还没有形成大规模关系图谱闭环。
- modifier fiber 目前主要来自 apple 微观因果图，还缺三模型复核。
- 当前 system atlas 仍然是语言系统的编码骨架，不是最终严格闭式数学。
