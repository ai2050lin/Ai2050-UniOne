## 总体结论

### 1. 组件级因果信号已经出现

- 这轮测试不是只看相关性，而是直接零化 `attn`（注意力）或 `mlp`（多层感知机）模块。
- 如果某一类组件被零化后，`margin`（正确候选与错误候选的平均对数概率差）系统性下降，说明该组件对当前能力具有更强因果贡献。

### 2. `qwen3 / glm4` 更偏 MLP 主导

- `qwen3`
  - `syntax`: `mlp_margin_drop = 1.1547`，略高于 `attn = 1.1137`
  - `relation`: `mlp = 1.9001` 明显高于 `attn = 1.1604`
- `glm4`
  - `syntax`: `mlp = 1.1471` 高于 `attn = 0.4771`
  - `relation`: `mlp = 1.5119` 高于 `attn = 0.7151`
  - `coref`: `mlp = 1.4346` 高于 `attn = 0.8035`

这说明在这两个模型里，当前三类判别任务更像“由 MLP 强化和稳定读出”的能力。

### 3. `deepseek7b` 更偏 Attention 主导，尤其在 relation / coref

- `deepseek7b`
  - `syntax`: `attn = 1.1766`，`mlp = 1.0117`，两者接近
  - `relation`: `attn = 1.2391` 远高于 `mlp = 0.2163`
  - `coref`: `attn = 0.8410`，而 `mlp = -0.6707`

这说明 `deepseek7b` 的关系和指代更依赖注意力路径。

### 4. `gemma4` 的组件因果结构仍然不稳定

- `gemma4`
  - 基线已经偏弱，`syntax baseline_pair_accuracy = 0.0`
  - 多个任务中出现负的 `margin_drop`
  - 这说明零化某些组件后，模型反而可能短暂改善结果

这里不能简单解释成“组件没用”，更合理的理解是：

- 基线本身不稳定
- 任务信号较弱
- 高旋转低放大导致局部零化结果更容易受随机波动影响

## 分能力结论

### syntax

- `qwen3`: `attn` 与 `mlp` 接近，`mlp` 略强
- `deepseek7b`: 二者接近，`attn` 略强
- `glm4`: `mlp` 明显更强
- `gemma4`: 基线过弱，不适合做强结论

### relation

- `qwen3`: `mlp` 更强
- `deepseek7b`: `attn` 更强
- `glm4`: `mlp` 更强

这说明“关系抽取”不是单一组件定律，而更像：

- 不同模型在不同组件上实现同一能力

### coref

- `qwen3`: 基线本身不够稳，信号弱
- `deepseek7b`: `attn` 更强
- `glm4`: `mlp` 更强
- `gemma4`: `attn` 略强，但整体不稳

这说明“指代消解”是当前三类任务里最复杂的一类。

## 第二阶段最重要的理论更新

1. 变量体系不只是相关性的。
   - 组件零化会系统性改变 `margin`，说明这些能力背后的确存在可破坏的局部机制。

2. 统一变量不等于统一组件实现。
   - `h_l / delta_l / d_l` 可能是跨模型共享的状态变量。
   - 但实现这些变量传播的主导组件，可以是：
     - `qwen3 / glm4` 偏 `mlp`
     - `deepseek7b` 偏 `attn`

3. 这反而让理论更清楚。
   - 真正共享的可能不是“哪类组件负责能力”，而是“能力必须经过某种差异方向放大与读出过程”。

## 当前最严格的表述

- 现在最稳妥的说法是：
  - **跨模型共享的是编码动力学变量，不一定是组件级实现细节。**
- 也就是说：
  - 同一种语言能力可以由不同模型通过不同组件分工实现
  - 但最后都落到相似的差异方向放大与读出框架上

## 下一步建议

1. 做 `stage640`
   - 在 `syntax / relation / coref` 上加入方向注入实验
   - 看“只改差异方向”能否恢复被消融组件造成的损失

2. 把 `coref` 单独加强
   - 当前它比 `syntax / relation` 更不稳定
   - 需要更多样本和更干净的句式设计

3. 明确理论边界
   - 现在可以较稳地说：
     - 共享变量成立
     - 共享组件不成立
   - 这是推进到第一性原理前非常关键的一步
