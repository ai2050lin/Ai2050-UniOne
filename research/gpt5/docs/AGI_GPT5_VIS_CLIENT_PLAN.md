# AGI GPT5 可视化客户端修改方案

## 1. 方案依据

根据当前项目状态，客户端不能再只强调“研究在推进”，而必须同时强调：

1. 研究进度
2. 证据链完整度
3. 判伪压力
4. 理论风险
5. 时间演化过程

原因很直接：

- 当前项目已经形成较强的统一解释骨架
- 但严格审查结果表明，理论还没有被强证据锁死
- 所以客户端的目标不应只是“展示理论很强”
- 而应升级为“展示项目到底推进了什么、站住了什么、还没站住什么”

换句话说，客户端需要从“研究展示页”升级成“研究驾驶舱”。

---

## 2. 对现状的判断

当前前端已经有不少可复用基础：

1. `App.jsx`
   - 已经是总入口
   - 同时挂了 DNN、SNN、ICSPB、3D 场景、进度面板、分析面板

2. `AGIProgressDashboard.jsx`
   - 已经有阶段路线图、时间线、路线对比、周报等基础进度视图

3. `ResearchProgressTab.jsx`
   - 已经有“理论 / DNN 分析 / 工程实现 / 里程碑”的大块结构

4. `RouteTimelineBoard.jsx`
   - 已经有按路线筛选、状态筛选、失败原因聚合、时间线导出

5. `AppleNeuron3DTab.jsx`
   - 已经有很强的 3D 研究工作台基础
   - 适合作为“机制层可视化”的主舞台

但当前也有明显问题：

1. 信息架构偏“面板堆叠”
   - 进度、理论、机制、风险、导入资产混在一起

2. 风险与判伪不在主视觉中心
   - 当前更容易看到“做了什么”
   - 不容易第一眼看到“哪里还不成立”

3. 文档进展、运行时间线、严格审查结果还没统一进一个入口

4. 2D 治理面板和 3D 机制场景之间缺少清晰分工
   - 现在很多东西都想放进 3D
   - 反而不利于判断研究状态

---

## 3. 核心改造目标

客户端改造后，用户一进入页面，应能在 `30` 秒内回答这 `5` 个问题：

1. 当前项目整体走到哪一步？
2. 当前最强进展是什么？
3. 当前最大硬伤是什么？
4. 当前理论为什么还不能算“成立”？
5. 接下来最该做哪一块？

因此客户端的主目标应定义为：

**把“进展、证据、风险、判伪、机制”五条线统一可视化。**

---

## 4. 新的信息架构

建议把客户端主入口改成 `5` 个一级视图：

### 4.1 总览驾驶舱

作用：

- 一屏看清项目状态
- 先给结论，再给细节

建议内容：

1. 项目综合进度
2. 严格审查状态
3. 当前最强成果
4. 当前最大风险
5. 最近一次关键阶段
6. 下一步任务块

建议视觉：

- 左侧：总进度 + 严格可信度双轴卡片
- 中间：路线热力矩阵
- 右侧：当前硬伤与下一步清单

### 4.2 路线与阶段中心

作用：

- 看阶段推进链和时间演化

建议内容：

1. Stage 时间线
2. 路线 A/B 对照
3. 失败原因聚合
4. 每阶段关键指标变化
5. 里程碑完成状态

建议视觉：

- 主体使用 `2D`
- 按路线分泳道
- 每个阶段节点支持展开“指标 / 测试 / 文档记录”

### 4.3 证据链与判伪中心

作用：

- 这是当前最缺的部分
- 专门回答“为什么现在还不能说理论正确”

建议内容：

1. 证据依赖图
2. 判伪边界面板
3. 最优律稳健性比较
4. 测试强度面板
5. 文档口径一致性检查

建议视觉：

- 依赖图用 DAG（有向无环图）或桑基图
- 右侧固定显示：
  - 当前是否存在自构造判伪
  - 当前测试强度是否过弱
  - 当前结论是否容易翻转

### 4.4 机制 3D 工作台

作用：

- 保留当前最强的 3D 机制展示能力
- 但只承载“机制层”问题

建议内容：

1. 神经元 / 路由 / 属性纤维 / 协议桥 3D 场景
2. 当前理论对象切换
3. 当前动作模式切换
4. 证据源联动高亮

建议原则：

- 3D 只负责“看机制”
- 不负责承担项目总治理信息

### 4.5 文档与日志回放中心

作用：

- 统一连接 `MEMO / MEMO2 / 严格审查 / 时间线`

建议内容：

1. 最新研究记录流
2. 关键节点对比
3. 文档差异对照
4. 从文档跳到对应阶段、测试、图表

## 5. 建议新增的关键可视化

### 5.1 双轴总览卡

不是只显示“项目进度”，而是同时显示：

1. 研究推进度
2. 严格可信度

这两个数必须并排显示，避免用户把“推进”误读成“已成立”。

### 5.2 证据依赖图

当前项目最大的风险之一，是高层结论复用下层摘要分数。  
所以需要一个明确的“证据依赖图”，显示：

- `Stage71` 依赖哪些块
- `Stage82` 依赖哪些块
- 哪些结论是直接证据
- 哪些结论是回灌汇总

### 5.3 判伪风险板

固定显示四项：

1. 是否存在脚本内自构造判伪
2. 是否存在文档口径与代码口径不一致
3. 是否存在最优律领先过小
4. 是否存在测试只验证回写自检

### 5.4 最优律稳健性带状图

当前 `sqrt` 虽然领先，但领先幅度很小。  
需要一个视觉上很直观的对比图，显示：

- `sqrt`
- `log`
- `rational`

三者的：

- 当前分数
- 与第一名差距
- 是否处于脆弱区

### 5.5 时间泳道板

比普通列表更好。  
建议按以下泳道拆分：

1. 理论推进
2. 测试推进
3. 严格审查
4. 文档收束

这样可以清楚看到：某次“理论提升”之后，是否真的跟上了测试和审查。

## 6. 数据层改造建议

当前客户端已经能拿到：

1. `/agi/progress`
2. `/api/v1/experiments/timeline`
3. 周报与路线运行数据

但还缺 `3` 类核心数据：

### 6.1 严格审查汇总接口

建议新增：

- `/api/v1/research/audit/latest`

返回：

- 严格审查状态
- 证据独立性分
- 测试强度分
- 当前硬伤列表
- 对应阶段链接

### 6.2 文档摘要接口

建议新增：

- `/api/v1/research/memos/feed`

返回：

- `MEMO`
- `MEMO2`
- `STRICT_AUDIT`

的最新结构化摘要，而不是让前端直接解析长文档。

### 6.3 理论对象状态接口

建议新增：

- `/api/v1/research/theory-objects`

返回：

- `family_patch`
- `concept_section`
- `attribute_fiber`
- `protocol_bridge`

等对象的：

- 当前状态
- 指标
- 风险
- 对应 3D 高亮参数

## 7. 前端文件级改造建议

### 7.1 `frontend/src/App.jsx`

建议改造：

1. 增加一个明确的“研究驾驶舱”入口
2. 把当前过重的总控逻辑拆出去
3. 保留现有 3D 工作台，但把项目治理视图从这里抽离

### 7.2 `frontend/src/AGIProgressDashboard.jsx`

建议改造为驾驶舱容器，不再只是一个进度页。  
内部拆成：

1. 总览
2. 路线与阶段
3. 证据链与判伪
4. 文档回放

### 7.3 `frontend/src/blueprint/ResearchProgressTab.jsx`

建议增强：

1. 增加“当前硬伤”区
2. 增加“严格审查结果”区
3. 理论区和工程区之间加“证据支持度”桥接区

### 7.4 `frontend/src/components/evaluation/RouteTimelineBoard.jsx`

建议增强：

1. 改成泳道式时间线
2. 支持按“理论 / 测试 / 审查 / 文档”筛选
3. 支持阶段节点展开对应文档摘要

### 7.5 `frontend/src/blueprint/AppleNeuron3DTab.jsx`

建议保留为机制层主舞台，但加一个“研究对象联动栏”，让 3D 节点高亮与理论对象状态统一。

6. `MemoFeedPanel.jsx`
   - 文档与日志流

7. `StageSwimlaneBoard.jsx`
   - 分泳道阶段时间线

## 9. 分期实施建议

### 第一阶段：结构重排

目标：

- 不大改视觉，只先把信息架构理顺

内容：

1. 增加“研究驾驶舱”入口
2. 把总览、时间线、严格审查拆成独立区
3. 让 3D 工作台退出“总控页”

### 第二阶段：证据链可视化

目标：

- 把当前最缺的“证据链 / 判伪 / 风险”补进来

内容：

1. 增加严格审查面板
2. 增加证据依赖图
3. 增加最优律稳健性图

### 第三阶段：文档联动与回放

目标：

- 让文档、时间线、可视化互相跳转

内容：

1. 增加文档摘要流
2. 支持从阶段跳转到对应文档
3. 支持从文档跳回对应测试与图表

---

## 10. 我建议的最小落地版本

如果只做一轮最有价值的修改，我建议先做这 `4` 件事：

1. 新建“研究驾驶舱”页
2. 增加“进度 / 可信度”双轴总览
3. 增加“严格审查”面板
4. 把 `RouteTimelineBoard` 改成按“理论 / 测试 / 审查 / 文档”分泳道的时间线

这四步做完后，客户端就会从“很多研究组件的集合”，变成“能支撑项目管理和理论判断的真正主界面”。

---

## 11. 最终一句话方案

**客户端改造的方向，不是继续堆更多面板，而是把现有 3D 机制能力保留，同时把进度、证据、风险、判伪、时间演化统一成一个研究驾驶舱。**
