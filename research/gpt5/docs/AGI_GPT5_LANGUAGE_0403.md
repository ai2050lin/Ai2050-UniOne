# AGI_GPT5_LANGUAGE

## 1. 研究目标

当前最核心的判断是：

深度神经网络的语言能力，不像“背了一本词典”，更像在参数、残差流与回路拓扑里形成了一套可计算、可组合、可分层分工的隐藏数学结构。

这套结构同时承载：

- 语义
- 语法
- 指代
- 属性绑定
- 多义切换
- 最终读出

真正要破解的，不是某一个小任务，而是这套内部编码机制本身。

## 2. 当前最稳的统一框架

截至目前，最稳的统一框架是：

`route（路由） -> backbone（骨干） -> switch（切换） -> attribute（属性） -> bridge（桥接） -> readout（读出）`

可以先写成统一状态方程：

`h_{t}^{l+1} = h_{t}^{l} + R_l(x_{<=t}) + B_l(lemma_t) + S_l(lemma_t, context_t) + A_l(attr_t, context_t) + G_l(B_l, A_l, route_t) + O_l(h_t^l)`

对应含义：

- `R_l`：早层路由项，负责跨词元关系建立，尤其是代词、功能词、句法线索的提前路由。
- `B_l`：名词骨干项，负责公共名词底座与家族复用。
- `S_l`：词义切换项，在共享底座上完成多义切换。
- `A_l`：属性修饰项，负责颜色、味道、大小等属性方向写入。
- `G_l`：绑定桥接项，把对象骨干与属性修饰真正绑定到同一对象。
- `O_l`：读出项，把内部状态压成最终可输出答案。

当前最通俗的理解是：

- `attention（注意力）` 更像负责路由、搬运、跨词元配对。
- `MLP（多层感知机）` 更像负责局部重写、状态压缩、类别整合。
- `residual stream（残差流）` 更像共享工作记忆。

## 3. 核心拼图

### 3.1 代词与功能词：路由先行

当前最硬的因果证据仍然来自代词主线。

`DeepSeek7B` 已经出现稳定的早层稀疏头组骨架，核心是：

- `H:2:0`
- `H:2:3`
- `H:3:1`

现阶段最稳的机制解释仍然是：

- 第 2 层先完成路由
- 第 3 层再做整合

这说明语言编码机制不能只理解成“哪一层亮了”，而要理解成“哪一组回路承担了先路由后整合的职责”。

### 3.2 多义名词：共享底座加切换轴

多义名词主线当前最稳的结论是：

- 不是每个词义一整套独立编码
- 而是“共享名词底座 + 词义切换轴”

`apple（苹果）` 的水果义和品牌义，就是在共享底座上沿不同方向切换。

这解释了为什么模型既有表达力，又能保持效率：

- 大量参数复用
- 切换成本较低
- 不需要为每一种二义性组合单独建库

### 3.3 普通名词与多义词的区别

当前统一协议已经给出比较稳的区别：

- 普通名词更像同一语义盆地内部的上下文扰动。
- 多义词更像共享底座上的跨语义盆地切换。

平均高活跃神经元交并比结果：

- `Qwen3`：
  - 多义词跨义交并比约 `0.0892`
  - 普通名词上下文交并比约 `0.2959`
- `DeepSeek7B`：
  - 多义词跨义交并比约 `0.1202`
  - 普通名词上下文交并比约 `0.2846`

这说明多义词不是“普通上下文漂移的放大版”，而是更深层的结构切换。

### 3.4 苹果个案：前面共享，到了切换带迅速分裂

苹果目前是最适合深挖的个案，因为它同时满足：

- 共享底座明显
- 切换轴明显
- 水果义与品牌义对比清晰

当前最稳结果是：

- 全网络 `top 256` 高活跃神经元口径下：
  - `Qwen3` 复用神经元 `10`
  - `DeepSeek7B` 复用神经元 `10`
  - 两者水果义独有与品牌义独有都远多于复用部分
- 最强复用层都在 `L1`
  - `Qwen3`：复用 `23`
  - `DeepSeek7B`：复用 `19`
- 最敏感切换层分别是：
  - `Qwen3 -> L5`
  - `DeepSeek7B -> L2`

在最敏感切换层的单层口径下，两模型都出现：

- 复用神经元 `6`
- 水果义独有 `58`
- 品牌义独有 `58`

这说明苹果编码机制更像：

- 早层共享底座
- 进入切换敏感层后迅速分裂

也就是“前面共享，到了切换带迅速分叉”。

### 3.5 切换轴：不是单点开关，而是方向性子结构

当前最接近机制层的说法是：

切换轴不是一个孤立神经元，而是一组有方向性的分布式神经元子结构。它们共同定义“往水果义推”还是“往品牌义推”的方向。

当前证据支持：

- `Qwen3` 最敏感切换层在 `L5`
- `DeepSeek7B` 最敏感切换层在 `L2`
- 切换轴消融有真实因果效应，但强度还不够大，说明它更像分布式方向，而不是极少数单点开关

### 3.6 属性绑定：骨干加修饰加桥接

更大属性图谱已经说明：

- 绑定不是整块重写
- 而是“名词骨干 + 属性修饰 + 桥接项”

也就是说，`apple-red（红苹果）`、`apple-sweet（甜苹果）`、`apple-fist-sized（拳头大小的苹果）` 这类复合概念，更像在对象骨干上叠加修饰通道，再由桥接项把两者绑定到同一对象。

现阶段较稳的结论是：

- `B_l` 和 `A_l` 已经有较强证据
- `G_l` 也就是桥接项，仍然是全理论里最薄弱的一块

## 4. 当前最硬的结论

截至目前，可以相对有把握地保留四条核心结论。

### 4.1 语言编码不是静态词典，而是分层状态演化

模型内部更像一个分层状态系统，而不是简单的词到标签映射。

### 4.2 多义词与普通名词不是同一种变化

普通名词更像同一语义盆地里的扰动；多义词更像共享底座上的结构切换。

### 4.3 苹果的水果义和品牌义并不是两套完全独立系统

它们共享早层底座，但在切换敏感层迅速分裂。

### 4.4 属性绑定更像组合，不像整块重建

对象、属性、桥接三者共同形成复合概念，这比“每个复合概念单独存一份表示”更符合当前证据。

## 5. 当前还没有解决的问题

要非常严格地说，当前理论还远没有到第一性原理。

最主要的硬伤有五个：

- `G_l` 也就是桥接项仍然偏弱，跨模型闭环还不完整。
- 神经元数量统计依赖阈值定义，比如 `top 256`、`top 64`，不是绝对自然常数。
- 部分因果结果仍受 `CUDA（显卡计算）` 不稳定影响，当前有些稳结论主要来自 `CPU（中央处理器）` 路径。
- 家族级切换轴因果效应还不够强，说明我们还没打中最小切换子回路。
- 当前方程还是中层机制理论，还不是可直接预测全部语言行为的第一性原理方程。

## 6. 目前最合理的中层解释

如果先不用过强的理论语言，当前最合理的中层解释是：

深度神经网络在语言处理中，学到的不是一张平面词表，而是一套分层组织的动态编码系统：

- 早层负责路由和共享底座搭建
- 中层负责语义切换、属性写入与关系重组
- 晚层负责整合与读出

其中：

- 功能词更容易露出稀疏回路骨架
- 多义名词更容易露出共享底座与切换轴
- 属性绑定更像分布式桥接场

这三条线目前已经可以互相拼接，但还没有完全闭合。

## 7. 下一阶段的大任务

如果目标是逼近第一性原理，下一步不该再停留在零散补实验，而应该直接推进四个阶段性大任务。

### 7.1 切换轴最小因果子回路搜索

围绕 `Qwen3 L5` 和 `DeepSeek7B L2`，搜索最小切换神经元组合，而不再只做方向级消融。

### 7.2 多阈值与累计口径协议

把当前神经元复用统计从单一 `top-k（顶部-k）` 口径，升级成多阈值、多层累计口径，避免定义过粗。

### 7.3 桥接项的回路化验证

把 `attention`、`MLP` 和残差方向一起纳入桥接搜索，验证 `G_l` 是否真能压缩成稳定小回路，还是天然更分布式。

### 7.4 统一动态理论闭环

把代词路由、多义切换、属性绑定三条主线，统一到同一个动态流程里，逼近真正可解释、可预测的语言编码状态方程。

## 8. 当前文档的定位

这份文档不再追求罗列所有实验，而只保留当前最核心的理论框架、关键拼图、最硬结论、主要硬伤与下一阶段主线。

它是当前研究进展的“最小理论骨架”，后续所有新实验都应该优先回答一个问题：

它是在强化 `R_l / B_l / S_l / A_l / G_l / O_l` 的哪一项，还是在推翻哪一项。

## 9. 苹果神经元在可视化客户端中的展示方案

### 9.1 设计目标

这次不是再做一个“苹果相关点云演示”，而是要把已经比较稳的研究结果压成一个可验证、可追因、可跨 layer 阅读的研究界面。

展示目标有四个：

- 展示哪些单元是真正有效的苹果切换部件，而不是只看激活值大。
- 展示这些单元在各个 `layer（层）` 中如何参与“共享底座 -> 切换 -> 放大 -> 读出”。
- 展示 `Qwen3` 和 `DeepSeek7B` 在同一抽象任务下的不同实现拓扑。
- 展示哪些结论可靠，哪些地方仍然有硬伤与不确定性。

### 9.2 应该接入的现有研究资产

客户端不应再凭手写假数据驱动，而应直接挂接以下四类现有结果：

- `stage438_apple_neuron_role_3d_view`
  - 提供苹果相关神经元的三类角色骨架：`noun_backbone（名词骨干） / sense_switch（词义切换） / attribute_modifier（属性修饰）`
- `stage448_apple_switch_layer_scan_and_neuron_counts`
  - 提供各层的共享/分裂计数、最敏感切换层、共享底座最强层
- `stage479_apple_switch_mixed_circuit_search`
  - 提供真正有因果效应的最小有效子回路
- `stage482_apple_switch_direction_tracking` 与 `stage484_apple_switch_signed_residual_basis`
  - 提供各层方向追踪、有符号切换推进、正向/反向峰值层、晚层平均有效性

如果只用 `stage438`，用户看到的只是“哪些点像苹果相关神经元”。

如果把 `stage479 + stage482 + stage484` 一起接入，用户才能看到“哪些点真的推动了苹果词义切换，以及这种推动在各层怎么传播”。

### 9.3 前端展示结构

建议直接挂到现有 `AppleNeuron3DTab` 主场景，不另起新页面。

页面结构分成四块：

2. 左侧筛选控制

- 义项切换：`fruit（水果义） / brand（品牌义） / compare（对比）`

  - 正向推进还是反向抵消

### 9.4 什么叫“有效神经元信息”

这次最重要的是定义展示口径，不能只按激活值排序。

建议把“有效神经元”定义成四级证据评分：

- `role_score（角色分数）`
  - 来自 `stage438`
  - 表示它像不像骨干、切换器、属性修饰器
- `causal_score（因果分数）`
  - 来自 `stage479`
  - 看单元或子回路 ablation（消融） 后，苹果切换任务掉了多少
- `signed_process_score（有符号过程分数）`
  - 来自 `stage484`
  - 看该单元在各层是沿切换轴正向推进，还是反向抵消
- `stability_score（稳定性分数）`
  - 来自 `stage485`
  - 标记该指标是否对 `CPU/GPU（中央处理器/图形处理器）` 敏感

最终前端显示的默认排序不要再是“最大激活值”，而应是：

`effective_score = 0.20 * role_score + 0.35 * causal_score + 0.35 * signed_process_score + 0.10 * stability_score`

这样做的原因很直接：

- `stage438` 解决“像什么”
- `stage479` 解决“有没有因果作用”
- `stage484` 解决“是正向推动还是反向抵消”
- `stage485` 解决“这个结论靠不靠谱”

### 9.5 苹果专题里建议重点展示的单元

当前最适合做首批展示的不是所有神经元，而是少量“最有解释力”的单元。

#### Qwen3

建议默认展示四个骨架头：

- `H:5:2`
- `H:5:29`
- `H:5:9`
- `H:5:8`

展示逻辑：

- 这四个头在 `stage479` 和 `stage484` 中都属于苹果切换主骨架
- 它们的晚层平均有符号耦合为负，说明消融后会削弱有效切换推进
- 但其顺向峰值并不都在最晚层，说明 `Qwen3` 更像“早中层写入，晚层读出放大”

#### DeepSeek7B

建议默认展示四个核心单元：

- `N:2:16785`
- `H:2:22`
- `H:2:10`
- `H:2:26`

其中前 3 个作为主骨架，`H:2:26` 单独标成“校正/争议单元”：

- `N:2:16785`
  - 是当前最关键的苹果切换锚点神经元
  - 在 `L2` 就高度敏感，并且 `pc1_explained_variance_ratio` 很高
- `H:2:22`
- `H:2:10`
  - 是主增强头
- `H:2:26`
  - 晚层有符号耦合为正
  - 更像分布校正或 heldout 增强头，不应和主切换骨架混成一类

### 9.6 3D 主场景怎么画

建议不要再像当前 `AppleNeuronCore3D` 那样按随机点生成网络，而要改成“研究驱动布局”。

每个点至少应包含：

- `unit_id`
- `model_key`
- `kind`
- `layer_index`
- `role`
- `effective_score`
- `causal_drop`
- `late_mean_signed_coupling`
- `forward_peak_layer`
- `reverse_peak_layer`
- `stability_flag`

布局建议：

- `x` 轴：真实 `layer index（层编号）`
- `y` 轴：角色带
  - 骨干
  - 切换
  - 属性
  - 混合
- `z` 轴：有效性强度或方向性

颜色建议：

- 蓝色：共享底座/骨干
- 橙色：正向切换支撑
- 红色：反向抵消或争议单元
- 青色：属性修饰
- 灰色：只在角色上显著、但因果不强

点大小建议由 `causal_score` 决定，而不是由原始激活值决定。

这样用户一眼看到的，就是“真正影响苹果词义切换的结构骨架”。

### 9.7 各个 layer 中对应的处理过程怎么展示

这是本次方案最关键的一块。

建议把 layer 过程拆成三条同步轨道：

1. 层级状态轨道
- 来自 `stage448`
- 展示每层：
  - 共享神经元数量
  - 水果义独有数量
  - 品牌义独有数量
  - `active_jaccard`
- 作用：
  - 告诉用户这层更像共享底座，还是已经强烈分裂

2. 因果影响轨道
- 来自 `stage479`
- 展示每个核心单元或子回路在各层相关任务上的破坏程度
- 作用：
  - 告诉用户哪些单元不是“相关”，而是“动它就坏”

3. 有符号传播轨道
- 来自 `stage482 + stage484`
- 展示每层：
  - `signed_contrast_switch_coupling`
  - `signed_pc1_switch_coupling`
  - `forward_peak_layer`
  - `reverse_peak_layer`
- 作用：
  - 告诉用户该单元是在把模型往正确词义推进，还是在往反方向推

对应到两模型的通俗解读应直接写在旁边：

- `Qwen3`
  - `L5` 是敏感切换层
  - 多个头从早中层开始写入
  - 最终在晚层读出时被统一放大
- `DeepSeek7B`
  - `L1` 更像共享底座最强层
  - `L2` 是切换锚定层
  - 后续多层更多是在传播、增强、校正

### 9.8 右侧详情卡片应该显示什么

当用户点击某个单元时，右侧应至少出现以下字段：

- 基本身份
  - `unit_id`
  - 模型
  - 层号
  - 类型
  - 当前角色
- 角色解释
  - 属于 `noun_backbone / sense_switch / attribute_modifier / mixed`
  - 为什么被分到这一类
- 因果解释
  - 消融后：
    - `search_drop`
    - `heldout_drop`
    - `control_abs_shift`
- 方向解释
  - 晚层平均有符号耦合是正还是负
  - 顺向峰值层在哪里
  - 反向峰值层在哪里
- 稳定性解释
  - 是否已有 `CPU/GPU` 对照
  - 峰值层位是否稳定

同时要给出一句自动生成的“研究摘要句”。

例如对 `N:2:16785`：

### 9.10 推荐实现路径

建议按三阶段做，不要一步全改。

第一阶段：先接数据，不大改 3D 结构

第二阶段：补 layer 过程时间线

### 9.11 严格审视下的问题、硬伤和瓶颈

这个方案虽然比现有苹果演示更接近研究真实结构，但仍有明显硬伤：

- `stage438` 的角色分类主要还是基于分数口径，不等于强因果证明。
- `stage479` 给出的是当前搜索协议下的最小有效回路，不保证是真正全局最小回路。
- `stage484` 的有符号结果已经很关键，但仍依赖当前切换轴定义；如果切换轴定义偏了，展示也会跟着偏。
- `CPU/GPU` 对峰值层位敏感，前端如果直接把峰值层画成绝对真相，会误导判断。
- 现阶段最强证据仍集中在 `apple`，如果直接做成“通用苹果理论成品”，会高估一般性。
