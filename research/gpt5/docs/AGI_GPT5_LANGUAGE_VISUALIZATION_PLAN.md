# AGI GPT5 语言主线客户端可视化方案

## 1. 方案目标

这份方案直接基于 `AGI_GPT5_LANGUAGE.md` 当前主结论整理，目标不是把客户端做成普通图表页，而是把语言主线里的核心结构压成一套可长期扩展的研究可视化系统。

客户端要优先回答 6 个问题：

1. 当前语言主线最稳的结构结论是什么。
2. 这些结论分别落在哪一层测试体系里。
3. 哪些现象已经稳定，哪些仍然只是候选结构。
4. 当前最大硬伤到底在哪里。
5. 不同模型与不同任务之间，哪些结论能迁移，哪些不能。
6. 下一阶段应该优先补哪一块拼图。

一句话目标：

**把“共享底盘 + 局部差分 + 路径放大 + 语义角色 + 来源保真缺口”做成可追踪、可对照、可回放的客户端研究驾驶舱。**

---

## 2. 视觉总原则

### 2.1 以研究对象为中心，不以图表类型为中心

客户端主角不应该是“柱状图、折线图、3D 图”，而应该是以下研究对象：

1. `共享基底`
2. `局部差分`
3. `路径放大`
4. `语义角色`
5. `来源保真`
6. `对象压力`
7. `跨模型迁移`
8. `跨任务主核`

也就是说，界面不按“图表种类”组织，而按“研究拼图”组织。

### 2.2 2D 负责判断，3D 负责解释

当前项目里最容易失焦的问题，是把太多治理信息塞进 3D。

建议明确分工：

1. `2D`
   - 看总览
   - 看阶段推进
   - 看风险
   - 看跨模型对照
   - 看指标收敛

2. `3D`
   - 看结构骨架
   - 看角色分布
   - 看路径切换
   - 看薄差分如何被放大

### 2.3 主视觉必须长期保留“硬伤”

根据语言主线，当前最硬的缺口不是前段触发，而是：

1. `天然来源保真低位`
2. `同类高竞争边界脆弱`
3. `跨模型硬主核偏少`
4. `结果角色弱于前段角色`

因此客户端不能只展示“亮点”，必须长期固定展示“当前硬伤板”。

---

## 3. 从语言主线到可视化对象的映射

### 3.1 五层测试体系，对应五块主视图

`AGI_GPT5_LANGUAGE.md` 已经把语言主线压成五层测试体系，客户端应直接按这五层做主导航：

1. `静态编码层`
   - 看共享基底、局部差分、差分厚度

2. `动态路径层`
   - 看任务词、方向词、操作词怎样切路径

3. `结果回收层`
   - 看原生闭合与修复闭合差异

4. `传播编码层`
   - 看来源痕迹怎样在传播中掉落

5. `语义角色层`
   - 看对象、属性、位置、操作、约束、结果怎样组成链

这样做的好处是：

- 客户端结构与研究文档口径一致
- 后续新增阶段更容易归档
- 用户不会在“机制图”和“进度图”之间失去上下文

### 3.2 当前最稳定结论，对应固定首页卡片

首页建议固定显示 5 张核心结论卡：

1. `共享基底强`
2. `局部差分薄`
3. `路径放大强`
4. `操作角色最稳`
5. `来源保真仍弱`

每张卡只保留三类信息：

1. 当前结论
2. 证据来源阶段
3. 当前风险说明

### 3.3 当前硬伤，对应固定风险区

固定风险区建议显示：

1. `品牌义边界弱`
2. `天然来源保真低`
3. `修复强于原生闭合`
4. `水果内部竞争最脆`
5. `跨模型硬主核仍少`

这块不应该折叠，也不应该埋到次级页面。

## 4. 建议的客户端信息架构

建议把客户端整理成 6 个一级视图。

### 4.1 总览驾驶舱

作用：

- 一屏看懂当前语言主线的总体状态

建议布局：

1. 左侧：`研究推进 / 证据强度 / 来源保真` 三轴卡
2. 中间：五层测试体系总览图
3. 右侧：当前硬伤与下一阶段任务

建议复用：

1. `AGIProgressDashboard.jsx`
2. `ProgressRiskDualAxis.jsx`
3. `StageSwimlaneBoard.jsx`
4. `TheoryAuditPanel.jsx`

新增内容：

1. `LanguageCoreSummaryPanel`
2. `HardProblemBoard`
3. `StageToLayerBridgePanel`

### 4.2 语义角色中心

作用：

- 把语言主线里的“角色先行”做成客户端主视图

建议展示：

1. 六类角色总图
2. 四类角色到六类角色的阶段演化
3. 角色强度排序
4. 角色到任务路径的桥

建议核心图层：

1. 角色矩阵图
2. 角色流向图
3. 角色到任务桥图
4. 角色稳定度时间线

核心目标：

**让用户一眼看出：语言不是词标签堆，而是角色组合系统。**

### 4.3 共享-差分-路径工作台

作用：

- 这是语言主线最核心的结构工作台

建议拆成三块并排区域：

1. `共享基底区`
   - 家族共享热图
   - 对象群聚关系

2. `局部差分区`
   - 差分厚度图
   - 高频差分位角色图

3. `路径放大区`
   - 小差分到大行为差异的路径图
   - 任务语义切入点图

建议核心交互：

1. 选择对象组
   - 水果
   - 动物
   - 工具
   - 品牌

2. 选择任务
   - 翻译
   - 修改
   - 重构

3. 选择模型
   - `GPT-2`
   - `Qwen`
   - `DeepSeek`

### 4.4 来源保真与闭合监测中心

作用：

- 专门显示当前最深瓶颈，不再把它隐藏在说明文字里

建议展示：

1. `天然来源保真` 走势
2. `原生闭合 vs 修复闭合` 对比
3. `高竞争对象组` 崩塌热区
4. 来源断裂点阶段图

建议视觉：

1. 上方用红橙色风险带
2. 中部用断裂链图
3. 下方用案例回放卡

核心目标：

**把“为什么前段成立，后段却掉”变成客户端里最直观的一页。**

### 4.5 跨模型与跨任务对照中心

作用：

- 避免单模型亮点被误读为硬主核

建议展示：

1. `GPT-2 / Qwen / DeepSeek` 对照矩阵
2. `翻译 / 图像编辑 / 代码重构` 跨任务主核筛选
3. 强迁移、弱迁移、不可迁移标签
4. 当前候选硬主核列表

建议核心图表：

1. 模型-任务热力矩阵
2. 迁移稳定性散点图
3. 主核保留漏斗图

### 4.6 阶段与文档回放中心

作用：

- 把 `Stage`、文档和图表打通

建议展示：

1. 阶段时间泳道
2. 文档摘要流
3. 某个结论对应的阶段证据链
4. 从文档跳到对应图表

建议复用：

1. `RouteTimelineBoard.jsx`
2. `MilestoneProgressPanel.jsx`
3. 现有 `MEMO / MEMO2` 摘要接口

## 5. 首页必须固定的指标体系

为了避免客户端只剩“漂亮图层”，建议首页固定保留以下指标组。

### 5.1 结构成立指标

1. `shared_base_mean`
2. `local_delta_mean`
3. `role_score`
4. `bridge_score`
5. `filter_score`

### 5.2 风险指标

1. `天然来源保真`
2. `同类高竞争失稳率`
3. `结果角色弱度`
4. `跨模型迁移衰减`

### 5.3 平台指标

1. 当前主分析模型
2. 当前主直测平台
3. 当前主瓶颈对象组

## 6. 数据模型建议

前端不要直接围绕零散接口写死，而应该先统一数据对象。

建议统一成 8 类前端实体：

1. `Stage`
   - 阶段编号、标题、摘要、时间、结论、风险

2. `Model`
   - 名称、参数规模、角色强项、瓶颈项

3. `Task`
   - 翻译、修改、重构、长链、高竞争

4. `Role`
   - 对象、属性、位置、操作、约束、结果

5. `ObjectGroup`
   - 水果、动物、工具、品牌、文件对象

6. `Metric`
   - 分数、均值、趋势、风险阈值

7. `Bridge`
   - 角色桥、任务桥、来源保真桥、路径放大桥

8. `Failure`
   - 断裂点、竞争失稳、迁移失败、结果回收失败

有了这层实体，前端才容易把不同页面联动起来。

## 7. 与现有前端的对接建议

### 7.1 优先复用，而不是重写

当前仓库已经有两类基础：

1. `AGIProgressDashboard.jsx` 这一类 2D 治理组件
2. `App.jsx`、`AppleNeuron3DTab.jsx`、`StructureAnalysisPanel.jsx` 这一类 3D 机制组件

因此最合理的路线不是推翻，而是补一层“语言主线专用容器”。

### 7.2 建议新增页面容器

建议新增：

1. `LanguageResearchDashboard.jsx`
2. `LanguageRoleCenter.jsx`
3. `LanguageStructureWorkbench.jsx`
4. `FidelityMonitoringCenter.jsx`
5. `CrossModelContrastCenter.jsx`
6. `StageDocumentReplayCenter.jsx`

### 7.3 建议新增公共组件

建议新增：

1. `LanguageCoreSummaryPanel`
2. `SharedDeltaPathPanel`
3. `SemanticRoleMapPanel`
4. `FidelityBreakdownPanel`
5. `CrossModelHeatmapPanel`
6. `HardProblemBoard`
7. `StageEvidenceBridgePanel`

### 7.4 语言配置要求

项目当前可切换语言，但默认显示中文，因此：

1. 新增页面默认中文
2. 关键英文术语保留中英双写
3. `locales.js` 补语言主线相关字段

## 8. 交互设计建议

### 8.1 研究对象选择优先于图层切换

不要先问用户“看热图还是看 3D”，而要先问：

1. 看哪个对象组
2. 看哪个任务
3. 看哪个模型
4. 看哪个阶段

随后再自动切换对应图层。

### 8.2 每个页面都要有“结论 / 证据 / 风险”三联结构

例如：

1. 顶部：当前结论
2. 中部：支持证据
3. 右侧或底部：当前风险

这样界面就不会滑向“只讲结果，不讲边界”。

### 8.3 允许从文档直接跳到图

文档里出现：

1. `Stage257`
2. `Stage258`
3. `Stage262`
4. `Stage265`

这些编号时，前端应支持一键定位到对应阶段图和指标图。

## 9. 分期实施建议

### 第一期：语言主线驾驶舱

目标：

- 先把语言主线的总览与硬伤做出来

内容：

1. 总览驾驶舱
2. 语义角色中心
3. 固定硬伤板

### 第二期：结构机制工作台

目标：

- 把共享-差分-路径主线做成主工作台

内容：

1. 共享基底图
2. 差分厚度图
3. 路径放大图
4. 任务入口桥图

### 第三期：来源保真与跨模型中心

目标：

- 把当前最难的两块拉到主视觉层

内容：

1. 来源断裂监测
2. 原生/修复闭合对照
3. 跨模型迁移矩阵
4. 跨任务主核筛选

### 第四期：阶段与文档联动

目标：

- 让研究记录、图表、阶段真正形成回放系统

内容：

1. 文档摘要流
2. 阶段泳道
3. 图文互跳

## 10. 最小可落地版本

如果只做一轮最有价值的客户端整理，我建议先落这 5 块：

1. `语言主线总览驾驶舱`
2. `语义角色总图`
3. `共享-差分-路径工作台`
4. `来源保真硬伤板`
5. `跨模型对照矩阵`

这样做完之后，客户端就能从“泛研究展示界面”升级成“语言主线研究中枢”。

---

## 11. 最终一句话方案

**基于当前语言主线，客户端最合理的方向不是继续堆通用图表，而是围绕“共享底盘、薄差分、路径放大、角色组合、来源保真缺口”五条主线，做成一套 2D 判断、3D 解释、文档可回放、阶段可追踪的研究可视化系统。**
