# AGI GPT5 整体修改方案

## 1. 总目标

当前项目的客户端不能只承担“展示研究成果”的功能。  
按照项目现在的真实状态，客户端必须同时承担 4 个职责：

1. 展示研究进度
2. 展示证据链
3. 展示风险与判伪
4. 展示机制层 3D 演示

所以整体修改目标不是“再加几个面板”，而是把前端整理成一个真正的：

**研究驾驶舱 + 机制工作台 + 文档回放系统**

---

## 2. 建议的项目结构

建议把前端结构按“角色”重新理解，而不是按“历史文件”理解。

### 2.1 顶层结构

建议保留并强化这几个目录职责：

1. `frontend/src`
   - 前端主入口与页面级容器

2. `frontend/src/components`
   - 通用组件
   - 包括进度、时间线、对比、共享卡片、工作台布局

3. `frontend/src/blueprint`
   - 蓝图级研究页面
   - 用于承载项目路线、系统状态、理论摘要、研究进展

4. `frontend/src/components/evaluation`
   - 研究治理层组件
   - 包括进度、严格审查、路线时间线、趋势、对照

5. `frontend/src/components/observation`
   - 观察层组件

6. `frontend/src/components/analysis`
   - 分析层组件

7. `frontend/src/components/intervention`
   - 干预与验证层组件

8. `frontend/src/components/shared`
   - 通用显示模板、卡片、历史、比较视图

9. `server`
   - 数据服务、运行时状态、研究摘要接口

10. `research/gpt5/docs`
   - 文档、计划、审查、备忘录

---

## 3. 建议的前端功能分层

建议把整个前端明确分成 `3` 层：

### 3.1 治理层

回答的问题：

- 项目走到哪一步？
- 哪条路线在推进？
- 当前最大风险是什么？
- 哪些结论还没有站稳？

对应页面：

- 研究驾驶舱
- 进度中心
- 严格审查中心
- 时间泳道

### 3.2 研究层

回答的问题：

- 理论主线是什么？
- 当前阶段的关键任务块是什么？
- 每条路线做了什么实验？

对应页面：

- 蓝图页
- 研究进展页
- 路线详情页
- 文档回放页

### 3.3 机制层

回答的问题：

- 理论对象如何在 3D 中被演示？
- 当前结构、路由、纤维、协议桥如何可视化？
- 选中某个对象后如何联动数据说明？

对应页面：

- AppleNeuron 3D 工作台
- Logit Lens / 流形 / 回路 / TDA / 纤维束等空间视图

---

## 4. 建议的主界面体系

建议把整个客户端整理成 `6` 个一级界面。

### 4.1 首页：研究驾驶舱

作用：

- 一进来先看全局，不先钻细节

应包含：

1. 项目综合进度
2. 严格可信度
3. 当前最强成果
4. 当前最大硬伤
5. 最近关键阶段
6. 下一步任务块

建议保留当前风格：

- 深色背景
- 卡片式分区
- 细边框
- 轻微色彩强调

### 4.2 路线与阶段中心

作用：

- 看研究推进链，而不是只看静态状态

应包含：

1. 阶段卡片
2. 路线时间线
3. 可行性趋势
4. A/B 路线对照
5. 里程碑状态

### 4.3 严格审查中心

作用：

- 专门展示“为什么现在还不能说理论成立”

应包含：

1. 理论可信度
2. 证据独立性
3. 测试强度
4. 风险标记
5. 审查发现列表
6. 下一步判伪建议

### 4.4 研究蓝图页

作用：

- 用结构化方式看理论、分析、工程、系统状态

对应当前已有能力：

- `roadmap`
- `analysis`
- `progress`
- `system`

建议不改风格，只强化边界：

- `roadmap` 讲主线和路线
- `analysis` 讲理论与深度分析
- `progress` 讲研究推进和阶段证据
- `system` 讲运行时状态与当前工程可用度

### 4.5 文档与日志页

作用：

- 连接 `MEMO / MEMO2 / STRICT_AUDIT`

应包含：

1. 最新文档摘要流
2. 关键节点记录
3. 当前阶段相关文档
4. 日志与阶段时间线联动

### 4.6 3D 机制工作台

作用：

- 专门做机制演示，不承担项目治理

应包含：

1. 理论对象切换
2. 分析动作切换
3. 3D 主场景
4. 选中对象详情
5. 数据资产联动

---

## 5. 每个界面的功能定义

下面给出每个界面的清晰功能边界。

### 5.1 研究驾驶舱

核心功能：

1. 双轴总览
   - 研究推进度
   - 严格可信度

2. 当前风险摘要
   - 自构造判伪
   - 最优律脆弱
   - 文档口径不一致
   - 测试强度不足

3. 最近阶段摘要
   - 最近 3 到 5 个关键阶段

4. 下一步任务
   - 直接给出接下来最值得做的块

### 5.2 路线与阶段中心

核心功能：

1. 路线总览
2. 各路线进度
3. 时间线筛选
4. 失败原因聚合
5. 按分析类型过滤
6. 导出阶段数据

### 5.3 严格审查中心

核心功能：

1. 审查分数面板
2. 证据依赖风险
3. 判伪真实性检查
4. 最优律稳健性
5. 当前不能成立的原因
6. 未来该怎么补证据

### 5.4 蓝图页

核心功能：

1. 展示理论路线
2. 展示工程路线
3. 展示系统状态
4. 展示研究进展
5. 作为项目主叙事界面

### 5.5 文档与日志页

核心功能：

1. 看最新文档记录
2. 按日期回放研究进展
3. 从文档跳转到路线与阶段
4. 从阶段跳回文档证据

### 5.6 3D 工作台

核心功能：

1. 演示理论对象
2. 演示结构机制
3. 演示动态路径
4. 绑定数据说明
5. 作为“机制层讲解器”

## 6. 3D 空间应该如何演示

这是当前最关键的部分之一。  
原则是：

**3D 空间用来演示“机制”，不用来承载“项目治理”。**

也就是说：

- 进度、风险、审查、时间线主要放在 2D
- 结构、路径、纤维、对象、关系、状态迁移放在 3D

---

## 7. 3D 空间的演示分层

建议把 3D 演示明确拆成 `4` 层。

### 7.1 结构层

演示内容：

- 节点
- 层级
- 群落
- 核心簇

适合演示对象：

- `family patch`
- `concept section`
- `restricted readout`

用户看到的效果：

- 哪些结构是底座
- 哪些节点是核心
- 哪些区域是高密度结构区

### 7.2 关系层

演示内容：

- 路由
- 连接
- 纤维
- 跨层关系

适合演示对象：

- `attribute fiber`
- `relation-context fiber`
- `stage-conditioned transport`

用户看到的效果：

- 信息如何跨区传播
- 哪些关系是稳定的
- 哪些路径是高频主路

### 7.3 动态层

演示内容：

- 路径亮起
- 节点激活
- 反事实分叉
- 路由切换

适合演示对象：

- `successor-aligned transport`
- `protocol bridge`
- `counterfactual`

用户看到的效果：

- 一个状态怎样进入下一个状态
- 什么情况下路径会分叉
- 当前闭环是否稳定

### 7.4 解释层

演示内容：

- 选中对象详情
- 当前对象对应的研究指标
- 当前对象对应的硬伤
- 当前对象与文档摘要的关系

用户看到的效果：

- 不是只看到炫的 3D
- 而是知道这个对象对项目意味着什么

---

## 8. 3D 空间的推荐交互方式

建议统一为以下交互链路：

1. 左侧选择理论对象
2. 顶部选择分析动作
3. 中间看 3D 演示
4. 右侧看选中对象详情
5. 底部看当前对象关联指标与风险

这样用户不会迷失。

建议交互动作：

1. 旋转
   - 看整体结构

2. 缩放
   - 看局部簇和路径

3. 选中
   - 看某个对象或节点的说明

4. 高亮
   - 联动相关路径与相关对象

5. 模式切换
   - 静态 / 动态 / 反事实 / 因果 / 组合性

---

## 9. 建议的 3D 演示脚本

建议每个理论对象都预置一个“演示脚本”，避免用户只是随便看。

### 9.1 `family patch`

演示脚本：

1. 先显示散点
2. 再显示簇形成
3. 再显示核心团块
4. 最后显示类别比较

讲解重点：

- 同类对象如何形成稳定底座

### 9.2 `concept section`

演示脚本：

1. 显示原型核
2. 显示实例偏移
3. 显示局部连续变化
4. 显示最小改动差分

讲解重点：

- 概念不是跳变，而是局部偏移

### 9.3 `attribute fiber`

演示脚本：

1. 显示维度簇
2. 显示属性方向
3. 显示维度切换
4. 显示组合叠加

讲解重点：

- 属性是否真沿纤维方向展开

### 9.4 `relation-context fiber`

演示脚本：

1. 显示关系点
2. 显示上下文主路
3. 显示跨层运输
4. 显示切换后的重组

讲解重点：

- 关系与语境如何传播并重组

### 9.5 `protocol bridge`

演示脚本：

1. 显示内部编码
2. 显示路由收束
3. 显示统一读出节点
4. 显示任务接口输出

讲解重点：

- 内部编码怎样进入可用输出

## 12. 最终推荐方案

如果只保留一句最核心的话：

**整体修改方案应该把项目客户端拆成“治理层 2D + 研究层蓝图 + 机制层 3D”三层系统，其中 3D 负责讲结构和路径，2D 负责讲进度、证据、风险和判伪。**

这样才能既保留当前项目已经很强的机制展示能力，又避免界面继续变成“什么都能放，但什么都不聚焦”的状态。
